## 2. 技术选型

### 2.1 核心框架对比

| 方案 | Electron | Tauri |
|-----|----------|-------|
| 包体积 | ~150MB | ~10MB |
| 内存占用 | 较高 | 较低 |
| 开发语言 | JS/TS | Rust + JS/TS |
| 生态成熟度 | 非常成熟 | 快速发展中 |
| Node.js 支持 | 原生支持 | 需额外集成 |
| 学习曲线 | 低 | 中等 |

**推荐选择：Electron**

理由：
1. 原生支持 Node.js，便于插件运行时实现
2. 生态成熟，社区资源丰富
3. 跨平台兼容性经过大量验证
4. 开发效率高，团队上手快

### 2.2 技术栈确定

| 层级 | 技术选择 |
|-----|---------|
| 桌面框架 | Electron 28+ |
| 前端框架 | React 18 + TypeScript |
| 状态管理 | Zustand |
| UI 组件库 | Radix UI + Tailwind CSS |
| 构建工具 | Vite + electron-builder |
| 插件通信 | IPC + JSON-RPC 2.0 |
| 数据存储 | SQLite (better-sqlite3) |
| 日志系统 | electron-log |

## 4. 核心模块设计

### 4.1 窗口管理

主窗口采用无边框设计，支持以下状态：
- **隐藏状态** - 默认状态，不占用任何资源
- **搜索模式** - 显示搜索框和结果列表
- **插件模式** - 展开显示插件 UI

### 4.2 插件运行时

#### Node.js 插件
- 使用 VM2 沙箱隔离执行
- 通过 IPC 与主进程通信
- 支持异步操作和 Promise

#### Python 插件
- 子进程方式运行
- 通过 stdin/stdout 进行 JSON-RPC 通信
- 内置轻量级 Python 运行时或使用系统 Python

### 4.3 数据存储

使用 SQLite 存储：
- 插件元信息
- 用户配置
- 使用统计
- 搜索历史
