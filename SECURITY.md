# Security Policy

## Supported Scope

Mulby 是一个具有高权限边界的桌面应用。安全问题不仅包括传统漏洞，也包括插件能力、命令执行、文件访问、网络访问、MCP 集成和 AI 工具权限相关的问题。

我们特别关注以下范围：

- 插件权限绕过
- Shell / 脚本执行限制绕过
- 文件系统访问越权
- HTTP / Network 范围限制绕过
- Git / Patch / Web Search / AI 工具能力越权
- MCP Server / Client 鉴权或工具暴露问题
- 自动更新链路、更新源、资源更新 manifest 验证问题
- 命令审计、来源校验、安装校验、`sha256` 校验缺陷
- 本地提权、任意代码执行、敏感数据泄露

不属于安全漏洞的情况通常包括：

- 需要用户主动安装并完全信任恶意插件后才成立的预期行为
- 纯文档错误、一般配置困惑、普通功能 Bug
- 没有越权、没有绕过、没有安全影响的实现缺陷

## Reporting a Vulnerability

请不要在公开 Issue 中直接披露安全漏洞细节。

建议通过以下方式私下报告：

- GitHub 账户主页可联系渠道
- 仓库维护者邮箱：`UnicellularSU@gmail.com`

邮件标题建议使用：

```text
[Mulby Security] 简短问题标题
```

报告中请尽量包含：

- 漏洞类型
- 影响范围
- 触发前提
- 复现步骤
- 预期行为与实际行为
- 影响版本
- 平台信息（macOS / Windows / Linux）
- 如有可能，提供 PoC、日志、截图或录像

如果问题涉及以下任一项，请明确写出：

- 是否需要安装插件
- 是否需要用户交互
- 是否需要特定设置或 Provider 配置
- 是否影响 `mulby-releases` 自动更新链路
- 是否影响外部 MCP 客户端或 Mulby 暴露的 MCP Server

## Response Expectations

收到报告后，维护者会尽快进行分流和确认。处理节奏会受问题复杂度、复现条件和平台差异影响，但目标是：

- 先确认是否能复现以及是否属于安全问题
- 再评估影响范围、修复方式和是否需要临时缓解措施
- 修复完成后，再决定公开披露方式

如果漏洞需要跨仓库修复，例如同时涉及 `mulby`、`mulby-cli`、`mulby-skills` 或发布链路，会统一协调处理。

## Disclosure Policy

在修复发布前，请避免公开披露可利用细节。

对于确认存在的问题，通常会按以下顺序处理：

1. 复现并确认影响
2. 准备修复
3. 发布修复版本或缓解方案
4. 在合适时机公开说明影响与修复结论

## User Guidance

在修复发布前后，用户侧的基本安全建议包括：

- 仅安装可信来源的插件
- 审慎启用高权限 AI 工具与命令执行能力
- 审慎连接外部 MCP 服务
- 对自动更新源、插件源和自定义 Provider 配置保持最小信任

Mulby 的能力设计本身偏向本地自动化与高自由度，这也意味着安全边界主要依赖权限控制、来源信任和最小授权原则。
